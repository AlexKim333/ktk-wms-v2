# ERPNext 표준 모듈 및 프레임워크 Q&A 기록

이 문서는 프로젝트 진행 중 발생한 ERPNext 기본 아키텍처 및 표준 모듈에 관한 질문과 답변을 기록한 문서입니다. 새로운 세션(또는 다른 기기)에서 AI(Antigravity)에게 이 문서를 읽게 하면 이전 대화의 맥락을 완벽하게 파악하고 이어서 대화를 진행할 수 있습니다.

## Q1. 고객 대상 판매와 지점 간 재고 이동 방식의 차이점
*   **질문:** 예약(견적서) 기능은 고객 대상인 것 같고, 지점/창고 간 재고 이동은 견적서 방식이 아닌 것 같은데 맞나요?
*   **답변 (네, 맞습니다):**
    *   **고객 대상 거래 (판매 프로세스):** `견적서(Quotation)` ➔ `판매 주문(Sales Order)` ➔ `출하(Delivery Note)`. 외부(고객)로 재고가 나가는 프로세스입니다.
    *   **내부 지점/창고 간 이동 (내부 물류 프로세스):** `자재 요청(Material Request - 목적: Transfer)` ➔ `재고 입력(Stock Entry - 목적: Transfer)`. 내부 이동이므로 견적서나 판매주문을 절대 사용하지 않습니다.

## Q2. 지점에서 창고로 재고 이동 요청 시 프로세스
*   **질문:** 지점에서 창고로 재고이동 요청을 하면 어떤 프로세스로 이루어지나요?
*   **답변:**
    *   **기본 프로세스 (즉시 이동):**
        1.  **지점 요청:** 지점에서 `Material Request (자재 이동)` 작성 및 제출 (상태: Pending)
        2.  **본사 출고 및 승인:** 본사 창고 관리자가 해당 요청서에서 `Stock Entry (재고 입력)` 문서 생성 및 승인(Submit)
        3.  **완료:** 승인 즉시 본사 재고 차감 및 지점 재고 증가 (요청서는 Completed로 자동 변경)
    *   **고급 프로세스 (운송 창고를 거치는 2단계 이동):** 물리적 배송 기간이 긴 경우 사용합니다. 본사 ➔ '운송 창고(가상)' ➔ '지점 창고'로 두 번에 나누어 재고를 이동(Stock Entry 2회 승인)하여 트럭 이동 중인 상태의 재고를 관리합니다.

## Q3. 지점의 백엔드 창고(Warehouse) 등록 여부
*   **질문:** 그렇다면 지점도 백엔드에서 창고로 등록해야 하나요?
*   **답변 (네, 필수입니다):**
    *   ERPNext에서는 재고를 보관하는 모든 논리적/물리적 공간을 반드시 '창고(Warehouse)' 마스터 데이터로 등록해야 합니다.
    *   **트리(Tree) 구조 구성:** `All Warehouses` 아래에 `본사 창고 그룹`과 `지점 창고 그룹`을 하위 폴더처럼 구성하는 것이 베스트 프랙티스입니다.
    *   **POS 판매 연동:** 매장 POS 판매 시 재고를 즉시 차감하려면, 지점 POS 프로필의 '기본 창고(Default Warehouse)'가 해당 지점 창고로 지정되어 있어야 합니다.

## Q4. POS 도입 전 지점을 창고로만 먼저 세팅하고 재고를 무한 누적시키는 전략
*   **질문:** 현재 각 지점 POS를 개발하기 전이라, 지점들을 창고로만 묶어두고 출고를 진행하여 재고가 무한정 쌓이게 둔 뒤, 나중에 POS 가동 시 '재고 조정'으로 맞추려는데 괜찮은가요?
*   **답변 (네, 훌륭한 단계별 도입 전략입니다):**
    *   초기에 본사 물류 흐름(Material Transfer)을 정상화하고 데이터를 안정적으로 구축하는 실무적인 베스트 프랙티스입니다.
    *   추후 POS 오픈 전, 물리적 실사(Physical Inventory)를 진행하고 `Stock Reconciliation (재고 조정)` 문서를 발행하여 시스템 누적 수량을 실제 수량으로 한 번에 덮어씌운 직후 POS 판매를 개시하면 완벽하게 연동됩니다. (이때 분실/로스도 한 번에 회계상 손실 처리됩니다.)

## Q5. 마진이 붙는 대리점(Franchise) 체제의 관리 방법
*   **질문:** 대리점은 마진이 붙어 나가는데 본사에서 지점처럼 관리하려면 어떻게 하나요?
*   **답변:** 대리점은 '지점(창고)'이 아닌 **'도매 고객(Customer)'**으로 취급하는 것이 원칙입니다.
    *   **표준 B2B 모델:** `Sales Order ➔ Sales Invoice` 프로세스를 태워야 본사 마진이 기록됩니다. 시스템 접근은 ERPNext의 'Customer Portal(고객 포털)' 기능을 활성화하면 충분합니다.
    *   **다중 회사(Multi-Company) 모델:** 본사가 대리점의 실재고/판매 내역까지 완벽히 통제하고 싶다면, ERPNext 내에 별도 Company를 세팅하여 권한을 완전히 분리하는 방식도 가능합니다.

## Q6. 프론트엔드(Vercel)와 백엔드(Frappe Cloud/AWS) 분리 배포 아키텍처
*   **질문:** 커스텀 앱 업데이트나 배포 시 일괄 적용이 가능한가요? Vercel과 프라페 클라우드의 역할은 무엇인가요?
*   **답변:**
    *   **백엔드 (Frappe Cloud / AWS 등):** 데이터베이스와 비즈니스 로직(ERP)을 담당합니다. Frappe 구조는 코드를 하나만 깔고 여러 개의 사이트(DB)가 공유하는 멀티 테넌시를 지원하므로, `bench update` 명령어 한 번에 100개의 대리점 DB가 일괄 업데이트됩니다.
    *   **프론트엔드 (Vercel):** 무거운 ERP 서버는 Vercel에 올릴 수 없습니다. 대신 개발 중인 가벼운 화면(Vue/React 기반 WMS/POS)을 Vercel에 배포하고, 백엔드 서버의 API를 호출하는 형태로 분리하는 것이 최적의 헤드리스(Headless) 아키텍처입니다.

## Q7. 한 컴퓨터에서 두 개의 아이디로 동시에 테스트(접속)하는 방법
*   **질문:** 브라우저 창을 2개 열어도 각각 다른 아이디로 로그인할 수가 없는데 어떻게 하나요?
*   **답변:** 일반 창은 세션을 공유하기 때문입니다. 분리하려면 1) **시크릿 모드** 창을 띄우거나, 2) 아예 **다른 브라우저**(크롬과 엣지)를 사용하거나, 3) **크롬 프로필 분리 기능**을 사용하여 독립적인 브라우저 환경을 만들어야 합니다.

## Q8. Custom Field (사용자 정의 필드) 추가 용량 한도
*   **질문:** 커스텀 필드를 몇 개 추가하지 않았는데 "Database Table Row Size Utilization" 게이지가 57%나 차 있습니다. 유료 플랜에 따른 한도인가요?
*   **답변 (아닙니다, 데이터베이스의 기술적 한계입니다):**
    *   Sales Invoice처럼 기본 필드가 이미 수백 개인 문서는 기본적으로 용량을 많이 차지합니다. MariaDB 자체의 '한 줄(Row) 최대 바이트' 제한 때문에 발생합니다.
    *   **해결책:** 텍스트 필드를 추가할 때 `Data` 대신 `Small Text` 나 `Text` 타입을 사용하면 게이지를 거의 차지하지 않습니다. 관련된 필드가 너무 많다면 아예 새로운 `Child Table (자식 테이블)`로 분리하여 연결하는 것이 가장 좋습니다.
