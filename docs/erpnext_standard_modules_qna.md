# ERPNext 표준 모듈 및 프레임워크 Q&A 기록

이 문서는 프로젝트 진행 중 발생한 ERPNext 기본 아키텍처 및 표준 모듈에 관한 질문과 답변을 기록한 문서입니다. 새로운 세션(또는 다른 기기)에서 AI(Antigravity)에게 이 문서를 읽게 하면 이전 대화의 맥락을 완벽하게 파악하고 이어서 대화를 진행할 수 있습니다.

## Q1. 고객 대상 판매와 지점 간 재고 이동 방식의 차이점
*   **질문:** 예약(견적서) 기능은 고객 대상인 것 같고, 지점/창고 간 재고 이동은 견적서 방식이 아닌 것 같은데 맞나요?
*   **답변 (네, 맞습니다):**
    *   **고객 대상 거래 (판매 프로세스):** `견적서(Quotation)` ➔ `판매 주문(Sales Order)` ➔ `출하(Delivery Note)`. 외부(고객)로 재고가 나가는 프로세스입니다.
    *   **내부 지점/창고 간 이동 (내부 물류 프로세스):** `자재 요청(Material Request - 목적: Transfer)` ➔ `재고 입력(Stock Entry - 목적: Transfer)`. 내부 이동이므로 견적서나 판매주문을 절대 사용하지 않습니다.

## Q2. 지점에서 창고로 재고 이동 요청 시 프로세스
*   **질문:** 지점에서 창고로 재고이동 요청을 하면 어떤 프로세스로 이루어지나요?
*   **답변:**
    *   **기본 프로세스 (즉시 이동):**
        1.  **지점 요청:** 지점에서 `Material Request (자재 이동)` 작성 및 제출 (상태: Pending)
        2.  **본사 출고 및 승인:** 본사 창고 관리자가 해당 요청서에서 `Stock Entry (재고 입력)` 문서 생성 및 승인(Submit)
        3.  **완료:** 승인 즉시 본사 재고 차감 및 지점 재고 증가 (요청서는 Completed로 자동 변경)
    *   **고급 프로세스 (운송 창고를 거치는 2단계 이동):** 물리적 배송 기간이 긴 경우 사용합니다. 본사 ➔ '운송 창고(가상)' ➔ '지점 창고'로 두 번에 나누어 재고를 이동(Stock Entry 2회 승인)하여 트럭 이동 중인 상태의 재고를 관리합니다.

## Q3. 지점의 백엔드 창고(Warehouse) 등록 여부
*   **질문:** 그렇다면 지점도 백엔드에서 창고로 등록해야 하나요?
*   **답변 (네, 필수입니다):**
    *   ERPNext에서는 재고를 보관하는 모든 논리적/물리적 공간을 반드시 '창고(Warehouse)' 마스터 데이터로 등록해야 합니다.
    *   **트리(Tree) 구조 구성:** `All Warehouses` 아래에 `본사 창고 그룹`과 `지점 창고 그룹`을 하위 폴더처럼 구성하는 것이 베스트 프랙티스입니다.
    *   **POS 판매 연동:** 매장 POS 판매 시 재고를 즉시 차감하려면, 지점 POS 프로필의 '기본 창고(Default Warehouse)'가 해당 지점 창고로 지정되어 있어야 합니다.
